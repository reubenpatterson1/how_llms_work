# Module 3: Context & Sessions — Design Spec

**Date:** 2026-04-22
**Status:** Approved design, pending implementation plan
**Author:** Reuben Patterson (w/ Claude)

## 1. Summary

A new interactive module for the LLM Engineering Course, inserted as **Module 3** (pushing current Modules 3–6 to 4–7). Topic: **Context vs. Prompting**, with the balance of prompt and context engineering as the two levers on the same attention architecture. The module's narrative spine is **"lost-in-the-middle"** — led by the Liu et al. 2023 finding, grounded in the attention math from Modules 1–2, and extended into practical session-hygiene and context-governance practices.

The module ships as a standalone React/Vite app (`context-and-sessions/`), matching the pattern of Module 2 (`working-with-llms/`). It contains 23 slides, 5 interactive components (including 3D visualizations powered by `@react-three/fiber`), and a deep technical assessment (15 MCQ + 4 diagnostic scenarios).

## 2. Goals and Non-Goals

**Goals**
- Teach lost-in-the-middle as a first-class concern, not a footnote
- Establish "context ≠ prompt" and make the distinction operational
- Give learners a hands-on feel for attention-budget allocation
- Equip learners with six session-hygiene practices + governance fundamentals
- Parity with Modules 1 and 2 on interactivity: 3D animations, live-manipulable tools, deep assessment
- Preserve existing Module 1 and 2 recording links

**Non-Goals**
- RAG-specific content (belongs in a future module)
- Model training / fine-tuning
- Provider-specific pricing / rate-limit discussion (belongs in Deployment module)
- Cross-browser QA beyond modern Chromium (matches existing modules)

## 3. Narrative Spine

**Through-line:** *Lost-in-the-middle is real. Here's why (attention-budget math), here are the two levers (prompt vs. context engineering), and here's how to keep sessions healthy and governed.*

Flow:
1. **Hook** — "You've been lied to about context length"
2. **Finding** — Lost-in-the-middle (Liu et al. 2023)
3. **Why** — Callback to attention dilution from Module 1
4. **Anatomy** — What actually lives in a context window
5. **Budget framing** — Attention is 100%, must be allocated
6. **Two levers** — Prompt engineering ↔ context engineering on same mechanism
7. **Governance surface** — Context as a compliance/policy concern
8. **Hygiene practices** — Six practices + nuclear restart
9. **Assessment** — Deep technical + diagnostic scenarios

## 6. Component Specs

All components live in `src/components/`. Conventions follow Module 2.

### 6.1 `ContextWindowPlayground.jsx` (~600 LOC, 3D)

**Purpose:** Headline visual — learner physically sees the middle collapse.

**Visual:** 3D landscape in `<R3F.Canvas>`. X = token position, Y = attention weight, color = attention saturation.

**Interactions:**
- Window size slider: 1k → 200k tokens
- "Place needle" button: drag marker to any position; tooltip shows recall probability
- Model preset buttons: GPT-4 Turbo, Claude Sonnet, Gemini 1.5, generic-short-context
- Noise slider: adds distractor tokens

**Data source:** `src/data/window-playground-table.json` (precomputed by `scripts/generate-attention-tables.mjs`). Shape: `{model, window_size, position, noise_level} → recall_probability`.

### 6.2 `LostInTheMiddleCurve.jsx` (~450 LOC, 2D + 3D)

**Purpose:** Teach the canonical U-curve and show where it breaks.

**Visual:** Split view. Left: 2D recharts line for the U-curve. Right: 3D recharts surface over (position × context_length).

**Interactions:**
- Context length slider (4k → 200k)
- Noise level slider (0 → 1)
- Needle position slider
- "Compare to human memory" toggle: overlays a human serial-position curve

**Data source:** same lookup table as C1.

### 6.3 `AttentionBudgetAllocator.jsx` (~350 LOC, 2D live)

**Purpose:** "Context is a budget" — chunks compete for 100% of attention.

**Visual:** Horizontal stacked bar (100% wide) + sankey sketch; each chunk colored by type.

**Interactions:**
- Add chunk buttons: `+ System prompt`, `+ RAG doc`, `+ Tool output`, `+ Prior turn`, `+ User msg`
- Size slider per chunk
- Remove (×) per chunk
- Live display: "Attention per critical token: X%"

**Math:** live softmax-style compute with `useMemo`. Chunk weight = `softmax(chunk_tokens × model_attention_profile)`.

### 6.4 `LeverComparisonTool.jsx` (~500 LOC, split live)

**Purpose:** Make "two levers, same mechanism" concrete.

**Visual:** Two stacked panes. Each pane: task description (top), intervention toggles (middle), quality score bar 0–100 + sample output snippet (bottom).

**Interactions:**
- Left pane toggles (prompt engineering): XML structure, Role prime, Explicit constraints, Few-shot examples, CoT scaffolding, Output format spec
- Right pane toggles (context engineering): Front-load critical, Summarize history, Pin invariants, Prune tool outputs, Scope session, External memory
- "Match the other side" challenge mode

**Data source:** `src/data/lever-scores.json` — maps intervention sets to scores.

### 6.5 `SessionHygieneSimulator.jsx` (~700 LOC, replay + toggles)

**Purpose:** Module capstone. Teaches all hygiene practices through realistic replay.

**Visual:** Three vertical zones:
- Top (30%): scrolling chat transcript (25 turns); critical info at turn 2, noise through turns 5–20
- Middle (50%): 3D attention landscape showing recall across the session
- Bottom (20%): 8 intervention toggles (see below)

**Interventions:**
1. Front/tail-load critical
2. Summarize old turns
3. Pin invariants
4. Prune tool outputs
5. Scope this concern
6. Watch effective context
7. External memory — sub-panel with:
   - Load PII/confidential guardrail
   - Attach org API spec index
   - Attach policy/compliance index
8. Nuclear restart (with learned summary only)

**Scoring:** live recall metric for the turn-2 critical fact. Start state ~15%. Target ≥90%.

**Data:** `src/data/simulator-session.json` (transcript) + rules table for intervention effects.

**Reuse hook:** accepts a `preset={scenarioId}` prop for the assessment's diagnostic scenarios.

## 7. Assessment Spec

### `ContextSessionsAssessment.jsx` (~1100 LOC)

Pass threshold: ≥80% overall AND ≥75% on diagnostics.

### Part 1: 15 MCQ

Five per bucket:

- **Bucket A — Attention & Lost-in-the-Middle:** U-curve, primacy/recency, effective vs. advertised length, attention dilution math, softmax mechanics
- **Bucket B — Context vs. Prompt:** what counts as context, token costs, two-levers framing, lever identification
- **Bucket C — Session Hygiene & Governance:** when to restart, summarize, pin invariants; raw-tool-output pitfall; indices; PII guardrails

Each item: 4 options, 1 correct, optional prose rationale (rubric-based partial credit), immediate feedback with slide reference on submit.

### Part 2: 4 Diagnostic Scenarios

Each renders a preset mini-replay of `SessionHygieneSimulator`. Learner must:
1. Read the transcript
2. Identify the failure (MCQ)
3. Pick the fix (MCQ)
4. Apply the fix in the mini-simulator; hit ≥90% recall
5. Explain why (2–3 sentence prose)

| # | Failure pattern | Correct diagnosis | Correct fix |
|---|---|---|---|
| D1 | 30-turn session, critical constraint at turn 3, ignored by turn 25 | Lost in middle — no pinning | Pin invariants + tail-load |
| D2 | 4 raw tool-output dumps, ~8k tokens each | Tool output pollution | Prune tool outputs + scope |
| D3 | PII in user input leaking into output | No inbound governance | External memory: PII guardrail |
| D4 | Session mixing billing bug + new feature | Scope bleed | Scope + nuclear restart with summary |

### Results screen

- Per-bucket + overall score
- Downloadable PDF certificate (reuses existing cert pipeline from commit `e81a4fb`)
- Progress written to `localStorage.llm_course_progress.part3`
- "Weakest area: X — rewatch slide N" recommendation

### Data

- `src/data/assessment-questions.json`
- `src/data/diagnostic-scenarios.json`

### localStorage migrator (one-shot, idempotent)

Lives at the top of `render()` in the landing page:

```js
function migrateProgressV1toV2() {
  const raw = localStorage.getItem('llm_course_progress');
  if (!raw) return;
  let p;
  try { p = JSON.parse(raw); } catch { localStorage.removeItem('llm_course_progress'); return; }
  if (!p || p.__v === 2) return;
  const next = { __v: 2 };
  if (p.part1) next.part1 = p.part1;
  if (p.part2) next.part2 = p.part2;
  // no part3 carryover — Context & Sessions is new
  if (p.part3) next.part4 = p.part3;
  if (p.part4) next.part5 = p.part4;
  if (p.part5) next.part6 = p.part5;
  if (p.part6) next.part7 = p.part6;
  localStorage.setItem('llm_course_progress', JSON.stringify(next));
}
```

## 9. Testing Strategy

**Stack:** Vitest + `@testing-library/react`; Playwright for the one critical e2e flow.

**Covered:**
- Data integrity (all 4 JSON data files)
- Slide engine (unique ids, valid types, keyboard nav, progress write)
- Component smoke tests (render + one interaction each)
- Assessment grading (pure functions in `src/lib/assessment.js`)
- localStorage migrator (edge cases, idempotency, malformed JSON)
- Landing page renumber e2e (seed old progress, verify new render + mapping)

**Not covered:** 3D visual quality, Three.js perf benchmarks, recharts rendering, certificate pixel-matching, cross-browser (matches existing Modules 1/2 practice).

**Test layout:**
